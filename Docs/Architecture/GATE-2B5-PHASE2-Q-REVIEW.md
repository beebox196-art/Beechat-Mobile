## Blockers

| # | Issue | Impact | Fix |
|---|-------|--------|-----|
| B1 | `.sheet` + `.popover` on the same view with the same `isPresented` binding will **not** adaptively present as sheet on iPhone and popover on iPad. SwiftUI applies both modifiers — whichever is evaluated last wins, and on iPad both `.sheet` and `.popover` will compete. The result is undefined behavior: on some iOS versions the sheet takes precedence, on others the popover does, and neither cleanly adapts to size class. | The NewTopicSheet presentation is broken at the core — it either double-presents or only uses one presentation style regardless of device. | Use a single `.sheet(isPresented:)` with `.presentationDetents` for iPhone and detect iPad/regular size class to swap to `.popover` via `@Environment(\.horizontalSizeClass)`. Alternatively, use `popover(isPresented:attachmentAnchor:)` only and let SwiftUI handle adaptive behavior (on iOS 16.4+ popovers adapt to sheets on compact size class). Do NOT attach both `.sheet` and `.popover` to the same view with the same binding. |
| B2 | `archiveTopic(id:)` in the spec uses `persistenceStore.topicRepo.save(topic)` after mutating `topic.isArchived = true` on a **value type** copy from the `topics` array. But `Topic` is a struct — the line `topic.isArchived = true` modifies a local copy, and `save()` will use `upsertPreservingCreatedAt` which only updates columns in `upsertColumns`. The `upsertColumns` list includes `isArchived`, so this technically works. **However**, `TopicRepository` already has a dedicated `archive(topicId:)` method (line 59) that does a direct `UPDATE` SQL. The spec ignores it. | The spec's ViewModel code reinvents an existing repo method and the comment "No separate `TopicRepository.archive()` method needed" is factually wrong — the method already exists. Using `save()` instead of `archive()` also loses the simpler SQL path. | Rewrite `archiveTopic(id:)` to use `persistenceStore.topicRepo.archive(topicId: id)` instead of `save()`. Update the spec note to reflect that `archive()` already exists. |
| B3 | `importCandidates()` is `async throws` but the ViewModel is `@MainActor`. The method calls `syncBridge.fetchSessions()` which is also async. The issue: `syncBridge.fetchSessions()` is **not** `@MainActor`-isolated (SyncBridge is not marked `@MainActor`). Inside an `@MainActor` method, calling `await syncBridge.fetchSessions()` hops to the correct executor. This works. However, `persistenceStore.topicRepo.fetchAllActiveSessionKeys()` is **synchronous** and accesses the DB on the reader — but `TopicRepository` is not `@MainActor` and uses GRDB's reader which has its own queue. This is safe because GRDB's `DatabaseReader.read` dispatches to its own serial queue. **No actual blocker** — but the spec doesn't address the `@MainActor` crossing and a reader might assume these calls are MainActor-safe when they're actually fine due to GRDB's internal dispatching. | Misleading but not broken. | Add a comment in the spec clarifying that `fetchAllActiveSessionKeys()` is safe to call from `@MainActor` because GRDB's reader dispatches to its own queue. Not a compile blocker, but documentation debt. |
| B4 | The `importSelected(_:)` method creates a `Topic` with `sessionKey: session.id` but doesn't call `saveBridge()` to create the bridge entry. The spec code has `saveBridge(topicId:sessionKey:)` wrapped in a do/catch, but the bridge entry is critical — without it, `resolveTopicId(for:)` won't find the topic, and `sessionKey(for:)` in the ViewModel won't resolve the gateway session key. More critically, `fetchAllActiveWithCounts()` uses a SQL JOIN on `topic_session_bridge` — without a bridge row, the `messageCount` will always be 0 for imported topics even when they have messages. | Imported topics will show 0 message count and may fail to resolve session keys for message loading. The `saveBridge` call is present in the code but the do/catch silently skips on error with `continue`, meaning the topic is created but the bridge might fail silently. | This is actually fine — the `saveBridge` call IS in the spec code. But the `continue` on error means the topic exists without a bridge. Change the error handling: if `saveBridge` fails, the topic should also be rolled back (delete the topic), or at minimum log the error at warning level. Also: after importing, the message count will still be 0 until `syncMetadataFromSessions` runs — clarify this in the spec. |

## Warnings

| # | Issue | Severity | Fix |
|---|-------|----------|-----|
| W1 | **Exyte ChatView `.disabled()` on input bar:** The spec suggests `.disabled(viewModel.connectionState != .connected)` on `ChatView`. Exyte's `ChatView` has no `.disabled()` override — SwiftUI's built-in `.disabled()` would propagate to all subviews including the message list, making messages unscrollable. The correct approach is to use `.setAvailableInputs([.text])` or provide a custom `inputViewBuilder` that shows a disabled placeholder. Neither is addressed. | High — offline composer is a spec requirement and the proposed approach doesn't work. | Use a custom `inputViewBuilder` that conditionally shows either the normal input or a "Reconnect to send messages" placeholder. This requires using the `ChatView` init that takes `inputViewBuilder`. Alternatively, overlay the input area with a ZStack when offline and intercept touches. The spec's own comment acknowledges this needs Q's verification — mark it as a design decision to resolve before implementation. |
| W2 | **Toast implementation:** The spec's overlay toast uses `DispatchQueue.main.asyncAfter(deadline: .now() + 5)` for auto-dismiss. This doesn't account for the view being dismissed and recreated (e.g., navigation pop/push). If the user navigates away and returns, the toast state (`showArchiveUndo`, `archivedTopic`) persists but the timeout already fired. Also, `asyncAfter` doesn't cancel on view disappear — if the view disappears before 5s, the state is mutated on a stale view. | Medium — toast will occasionally misbehave during navigation. | Use a `Task` with `try await Task.sleep` instead of `DispatchQueue.main.asyncAfter`. Cancel the task in `.onDisappear`. Alternatively, use `withAnimation` + a `@State` timeout managed by SwiftUI's `task` modifier. |
| W3 | **`unarchiveTopic(id:)` revised version uses `fetchById()`** which returns `Topic?`. The spec creates the method as `try Topic.fetchOne(db, key: id)` inside `TopicRepository`. This is correct GRDB API (confirmed — `Topic` conforms to `FetchableRecord` via `UpsertableRecord`, and `id` is the primary key). However, the spec's `fetchById` returns `Topic?` but `fetchOne` returns `Topic?` only when using the `fetchOne(db, key:)` overload. The `fetchOne(db, key:)` method requires the type to have adopted GRDB's `FetchRequest` protocol, which `Topic` does via `TableRecord` + `FetchableRecord`. This should work. ✅ | Low — verified correct. | No fix needed. |
| W4 | **`TopicError` as nested vs. top-level:** The spec defines `TopicError` as `public enum TopicError: LocalizedError` at the top level of `BeeChatMobileViewModel.swift`. This is fine for a small codebase, but as the ViewModel grows, it mixes error types with state. Consider nesting it as `BeeChatMobileViewModel.TopicError` or moving it to a separate file. | Low — style preference. | Suggest nesting or extracting to a separate file. Not a build blocker. |
| W5 | **`fetchAllActiveSessionKeys()` queries `TopicSessionBridge` table** but the `Column("status") == "active"` filter uses a string literal. The `TopicSessionBridge` model defines `status` as a `String` property. This is fragile — if the status values change, this string will silently break. | Low — fragility. | Consider defining status constants (e.g., `TopicSessionBridge.Status.active`) or at minimum a string constant. Not blocking. |
| W6 | **`importCandidates()` filter logic:** The spec has a confusing comment block about `BeeChatSessionFilter.isBeeChatSession` and then falls back to `true` ("For now: show all sessions without topics"). This is the right MVP approach but the dead code comment is confusing. | Low — clarity. | Remove the commented-out filter logic and just have: `return candidates` with a `// TODO: Filter to user-created sessions only`. |
| W7 | **`createTopic(name:)` calls `topicRepo.create(name:pendingGatewaySync:)`** which generates a `sessionKey` internally as `"agent:main:\(topicId.lowercased())"`. The spec then checks `if !isOffline, let bridge = syncBridge, let sessionKey = topic.sessionKey` to send a bootstrap message. This is correct — `topicRepo.create()` sets the `sessionKey` before returning. ✅ | None — verified correct. | No fix needed. |
| W8 | **`archiveTopic(id:)` auto-selects first topic** after archiving the selected topic. But if the user archives a non-selected topic, it still refreshes `topics`. This is fine, but the `selectedTopicId` binding in `TopicListView` uses a `Binding` that sets `viewModel.selectedTopicId`. If the topic list shrinks and `selectedTopicId` still points to a valid topic, no re-selection happens. This is correct behavior. ✅ | None. | No fix needed. |
| W9 | **`ImportSessionsSheet` uses `List(candidates, id: \.id, selection: $selectedIds)`** for multi-select. SwiftUI `List` with `selection` binding only works for edit mode on iOS. The `selection` parameter on `List` is designed for macOS/iOS with `List(selection:)` in edit mode. On iOS, you typically need `@State var editMode = EditMode.active` or use `Toggle` per row. The `List(_:id:selection:)` with a `Set<String>` binding works for **macOS** but on iOS you need `.environment(\.editMode, .constant(.active))` or custom multi-select. | Medium — import sheet multi-select may not work on iOS. | Either: (a) add `.environment(\.editMode, .constant(.active))` to the List, or (b) use custom row views with `Toggle`/checkmarks. Test on iOS simulator before committing. |
| W10 | **Empty messages state in `BeeChatView`**: The spec adds `if messages.isEmpty && !viewModel.isStreaming { ... }` but doesn't specify where in the view hierarchy. The current `BeeChatView` wraps `ChatView` which takes the full space. Adding a conditional inside the `VStack` before/after `ChatView` won't overlay the empty state on the chat area. The empty state needs to be an overlay on the `ChatView` or replace the `ChatView` entirely when empty. | Medium — empty messages state won't display correctly. | Use `.overlay` on the `ChatView` to show the empty prompt, or use a `ZStack` with the empty state behind the `ChatView` (ChatView handles empty message arrays gracefully already — verify). |

## Questions

| # | Question | Context |
|---|----------|---------|
| Q1 | **Exyte ChatView offline composer:** What's the preferred approach for disabling the input bar when offline? Options: (a) custom `inputViewBuilder` showing a disabled placeholder, (b) overlay with tap-to-retry, (c) `showNetworkConnectionProblem(true)` which shows a "Waiting for network" bar (already built into Exyte). Option (c) is the lowest-effort but doesn't disable the input field itself. | The spec requires "Composer disabled or placeholder 'Reconnect to send messages' when offline." The simplest path is combining `showNetworkConnectionProblem(true)` with a custom input view builder. |
| Q2 | **Import sessions sheet multi-select:** Should we use iOS edit mode with `List(selection:)` or a custom checkmark-based approach? The edit mode approach is standard but requires activating edit mode. Custom checkmarks give more control but more code. | W9 above. Need design decision before implementation. |
| Q3 | **Archive undo scope:** Should the undo toast persist across navigation? If the user archives a topic and immediately taps another topic to open it, the toast disappears because `TopicListView` is still in the nav split. But if they pop back, the toast is gone. Is 5 seconds enough? | Current spec says 5 seconds. Consider 7 seconds with a cancel-on-new-action behavior. |
| Q4 | **`fetchById()` location:** Should this go in `TopicRepository.swift` in BeeChat-v5 or in `BeeChatPersistenceStore`? Currently `TopicRepository` is the right place, but the spec puts it in BeeChat-v5 which is a different package from the BeeChat-Mobile workspace. Need to confirm the repo file path. | The spec says "BeeChat-v5/Sources/BeeChatPersistence/Repositories/TopicRepository.swift" which is correct. |
| Q5 | **`importSelected` bridge failure handling:** When `saveBridge` fails for an imported topic (duplicate key), the spec does `continue` — skipping that session but creating the topic. This leaves an orphan topic with no bridge entry. Should we delete the topic on bridge failure, or is the topic usable without a bridge (via `topic.sessionKey`)? | The `Topic` model has `sessionKey` as a direct column, so messages can still be loaded via `sessionKey`. But `fetchAllActiveWithCounts()` uses bridge JOIN for `messageCount`. An orphan topic would show 0 messages even when it has them. |
| Q6 | **Empty messages state placement:** The spec says "inline in BeeChatView" but doesn't show where exactly. Should it be an overlay on the `ChatView`, a replacement for the `ChatView`, or inside the `VStack`? Exyte's `ChatView` already handles empty message arrays (shows an empty scrollable area). | Need to decide: overlay with custom prompt, or replace `ChatView` entirely with an empty state view when `messages.isEmpty && !isStreaming`. |
