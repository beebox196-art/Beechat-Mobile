# Gav Final Gate 2 Review: Dependencies & Prior Art

**Date:** 2026-05-15  
**Angle:** dependencies, package boundaries, avoiding wheel reinvention  
**Verdict:** Mostly right direction, but two issues should be fixed before Q builds.

## BLOCKER

### 1. Exyte/Chat is specified as pinned, but the manifests currently float

The spec says Gate 2 should pin Exyte/Chat at `2.7.10`. That is the right call for stability: GitHub shows `2.7.10` was tagged on 2026-04-09 and `3.1.0` was tagged on 2026-05-15, with the compare API reporting `3.1.0` is 38 commits ahead of `2.7.10`.

But the current mobile manifests use `from: 2.1.0`:

- `BeeChatMobile/Package.swift`
- `BeeChatMobile/project.yml`

That can resolve to `3.1.0`, directly contradicting the spec and risking API breakage in the first real pipeline build.

**Fix:** Change both manifests to an exact `2.7.10` requirement for Gate 2, and commit `Package.resolved`. Do not rely on the prose spec to enforce the pin.

### 2. Gate 2C optimistic-send code reinvents delivery state and will not compile

The spec's send-flow sample creates an optimistic `BeeChatPersistence.Message`, then later does:

```swift
optimisticMessage.status = .error
```

But v5 `BeeChatPersistence.Message` has no `status` field. Delivery status already lives in `BeeChatSyncBridge`'s delivery ledger (`pending`, `sent`, `delivered`, `failed`) and `SyncBridge.sendMessage` already creates the ledger entry with an idempotency key.

This is both a compile-time problem and a wheel-reinvention problem.

**Fix:** Do not add delivery status to `Message` for Gate 2. Keep optimistic UI state as UI/ViewModel state, or expose/read the existing delivery ledger if the UI needs durable status. For the basic pipeline, let `SyncBridge.sendMessage` own idempotency and ledger transitions.

## WARNING

### 3. The app target dependency boundary is over-corrected

The spec removed the app's direct dependency on `BeeChatMobileKit` and says the app accesses it through `BeeChatUI`. That only works if the app never needs to name `BeeChatMobileConfig`, `BeeChatMobileViewModel`, or other MobileKit types.

For a thin composition root, the app usually does need to load config/token and construct the root view model. Forcing that through `BeeChatUI` risks making the UI package own configuration and lifecycle concerns.

**Recommendation:** Either:

- allow the app target to depend directly on both `BeeChatUI` and `BeeChatMobileKit`, or
- expose a single `BeeChatRootView` from `BeeChatUI` that fully hides MobileKit types.

Do not use re-export tricks to paper over this. Direct app -> kit dependency is not harmful coupling when the app is the composition root.

### 4. Do not create an SPM executable target for the iOS app

The spec says to add a `BeeChatMobile` executable target. For iOS, keep the app as an Xcode application target generated by `project.yml`; use SPM for library targets only (`BeeChatMobileKit`, `BeeChatUI`). An SPM executable target is the wrong abstraction for the iOS app bundle, Info.plist, signing, simulator launch, and Xcode scheme flow.

**Fix:** Adjust the Package.swift section to say: local package provides library targets; Xcode app target depends on those products.

### 5. Exyte brings more transitive dependencies than the basic text pipeline needs

Exyte/Chat `2.7.10` requires `MediaPicker`, `ActivityIndicatorView`, `GiphyUISDK`, and `Kingfisher` in its Package.swift. The basic Gate 2 pipeline does not need Giphy, media picking, or image caching.

This is not enough reason to drop Exyte now because Gate 1 already proved the chat surface. But we should avoid adding any of those as direct dependencies and avoid building Gate 2 features around them.

**Gate 2 rule:** Direct external deps should be only Exyte/Chat and the v5 local package. Kingfisher, Giphy, MediaPicker, and ActivityIndicatorView are transitive implementation details of Exyte.

### 6. Exyte streaming/update risk is real; keep the coalescing requirement

There is an open Exyte issue titled "Fast message insertions or updates cause weird UI and too many view renders" (#223, opened 2025-09-11). That lines up with our streaming concern.

The spec already says to coalesce streaming updates. Keep that as a Gate 2 requirement, not an optional optimization. Avoid per-token full-array churn.

### 7. Use GRDB observation instead of manual refresh loops where practical

The spec says "no state duplication", but the ViewModel sketch has `messages` and `displayMessages` arrays plus manual `refreshMessages()` calls. v5 already uses GRDB `ValueObservation` patterns, and GRDB is already in the stack.

For Gate 2A/2B, prefer observing sessions/messages from GRDB and mapping to Exyte display models at the edge. Manual fetch/update loops are more likely to miss reconciliation updates and reintroduce bugs v5 already solved.

This can be scoped: do not build a generic repository framework. Just use GRDB observation for session list and current-session messages.

### 8. Swift/toolchain version in the dependency table is stale

The spec says Swift `5.9+`, while v5's `Package.swift` is `swift-tools-version: 6.0`, and GRDB `v7.10.0` declares `swift-tools-version: 6.1`. With Xcode 26 this is probably fine, but the spec should not imply Swift 5.9 is sufficient.

**Fix:** Say Xcode 26 / SwiftPM capable of tools 6.1. Keep targets in Swift 5 language mode where v5 already does that.

### 9. GRDB itself is not a stability concern for Gate 2

GRDB `v7.10.0` is current as of 2026-05-15 and its release notes focus on Android/Linux/Windows/SQLCipher work, not iOS regressions. Its Package.swift supports iOS 13+, so our iOS 17 floor is comfortable.

No extra persistence dependency is needed. Do not introduce SwiftData, SQLite.swift, or a wrapper around GRDB.

## NIT

### 10. Dependency table should list only direct dependencies

The current table lists Kingfisher as "External (SPM)". That is misleading unless Q adds it directly, which we should not do.

**Fix:** Split into:

- Direct: `Exyte/Chat` exact `2.7.10`, local `../../BeeChat-v5`
- Transitive via Exyte: `Kingfisher`, `GiphyUISDK`, `MediaPicker`, `ActivityIndicatorView`
- Transitive via v5: `GRDB`

### 11. Gate 2 does not need Valet or another Keychain wrapper

The spec is correct to keep raw Security/v5 `KeychainTokenStore` for now. Valet can wait until pairing/settings or iOS-specific Keychain code becomes nontrivial.

### 12. No WebSocket dependency should be added

The spec is correct: keep v5's `URLSessionWebSocketTask` transport. Starscream/Pusher/Socket.IO are not needed for Gate 2 and would duplicate protocol and reconnect logic that v5 already owns.

## Prior-Art Summary

Use what already exists:

- v5 owns WebSocket, RPC, reconnection, reconciliation, streaming state, idempotency, and delivery ledger.
- GRDB owns SQLite access and observation.
- Exyte owns chat list/input/bubble mechanics.
- Mobile should own only iOS configuration, lifecycle glue, Exyte mapping, and small UI states that do not belong in durable core data.

The biggest stability move is not adding anything new.
